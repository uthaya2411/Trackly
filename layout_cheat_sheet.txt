========================================================================
                     TRACKLY PRO - MASTER REFERENCE SHEET
========================================================================

This guide is split into three sections:
1. FLUTTER LAYOUTS: Building your premium visual UI.
2. RIVERPOD STATE: Sharing and controlling global app data.
3. FLUTTER LIFECYCLE: Writing robust, crash-free mobile code.

========================================================================
                      SECTION 1: FLUTTER LAYOUTS
========================================================================

1. SCAFFOLD (The Blank Canvas)
   - Think of this as the main drawing board for your screen.
   - It sets up the core screen background and handles structural parts 
     like app bars, navigation bars, and drawers.

2. STACK (The Layer Overlay)
   - Normally, widgets inside a Column or Row sit next to each other.
   - A Stack allows you to layer widgets ON TOP of each other (like placing 
     a transparent sheet of glass over a painting).

3. POSITIONED.FILL (The Background Wallpaper)
   - Used inside a Stack to stretch a widget (like a background color, 
     image, or gradient) to fill every single pixel of the screen.

4. RADIALGRADIENT (The Ambient Glow Effect)
   - Instead of a single boring flat color, this makes a soft circle of 
     purple/rose light that slowly fades out. 
   - This creates a signature "AMOLED Premium Dark Mode" aesthetic.

5. SAFEAREA (The Border Patrol)
   - Smartphone screens have camera notches, rounded corners, and swipe 
     indicators at the bottom.
   - SafeArea keeps your content inside the safe boundaries of the screen 
     so it doesn't get cut off.

6. SINGLECHILDSCROLLVIEW (The Overflow Shield)
   - On small phones, or when the on-screen keyboard pops up, your UI 
     can run out of height space.
   - This widget lets the user scroll your page up and down, preventing 
     the dreaded yellow-and-black stripe visual error blocks.


========================================================================
                      SECTION 2: RIVERPOD STATE
========================================================================

Riverpod is our "Central Bank Vault" for shared data. Instead of passing 
data manually from screen to screen, we store it globally. To access it, 
we use these specialized widgets instead of standard Flutter ones:

1. THE MAGIC KEY: "ref" (WidgetRef)
   - In standard Flutter, widgets only know about `BuildContext context`.
   - In Riverpod, widgets get `ref` (WidgetRef). This is your key to the 
     Central Bank Vault.
   - You use it to either WATCH data (for updating UI) or READ functions 
     (for buttons).

2. CONSUMERWIDGET (The Centralized Stateless Widget)
   - In standard Flutter, you use a `StatelessWidget` for simple screens.
   - In Riverpod, you use `ConsumerWidget` instead.
   - Inside its build method, it automatically gives you `WidgetRef ref`:
     `Widget build(BuildContext context, WidgetRef ref) { ... }`

3. CONSUMERSTATEFULWIDGET (The Centralized Stateful Widget)
   - In standard Flutter, you use `StatefulWidget` when you need local 
     variables (like text controllers or timers).
   - In Riverpod, you use `ConsumerStatefulWidget` instead.
   - It acts exactly like a normal stateful widget, but prepares your 
     class to talk to global providers.

4. CONSUMERSTATE (The Centralized State Room)
   - This is the partner to `ConsumerStatefulWidget`.
   - By inheriting from `ConsumerState` instead of normal `State`, the 
     magic `ref` key becomes globally available anywhere in your class!
   - You can write `ref.read(...)` inside `initState()`, `dispose()`, 
     or custom button tap functions easily.


========================================================================
                    SECTION 3: FLUTTER LIFECYCLE
========================================================================

1. MOUNTED (The Screen Life Detector)
   - A boolean value (`true`/`false`) built into every Flutter State.
   - Returns `true` if the screen is active and visible to the user.
   - Returns `false` if the screen has been closed, popped, or destroyed.
   - CRITICAL RULE: Always check `if (mounted)` before doing navigation 
     or using `context` after an asynchronous action (like saving data).
     This guarantees your app will NEVER crash if a user quickly exits 
     a page while it is loading/saving!
